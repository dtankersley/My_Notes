## 10.1 Network Configuration

### View IP Addresses

```
ip addr
```

Or shorter:

```
ip a
```

Output:

```
1: lo: <LOOPBACK,UP,LOWER_UP> ...
    inet 127.0.0.1/8 scope host lo
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> ...
    inet 192.168.1.105/24 brd 192.168.1.255 scope global dynamic eth0
```

- `lo` - Loopback interface (127.0.0.1 = localhost)
    
- `eth0` (or `ens33`, `enp0s3`) - Your network interface
    

### View Routing Table

```
ip route
```

Output:

```
default via 192.168.1.1 dev eth0
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.105
```

The default route (`default via 192.168.1.1`) is your gateway - where traffic goes to reach the internet.

### View and Set Hostname

```
hostname
```

hostnamectl

```

Shows detailed hostname and OS information.

### **The /etc/hosts File**

```

cat /etc/hosts

```

Output:

```

127.0.0.1 localhost 127.0.1.1 cato

```

This file maps hostnames to IP addresses locally. Useful for:

-   Adding shortcuts to servers you access often
    
-   Blocking websites (point them to 127.0.0.1)
    
-   Testing before DNS changes
    

Add an entry:

```

sudo vi /etc/hosts

```

Add:

```

192.168.1.50 myserver

```

Now `ping myserver` works without knowing the IP.

## **10.2 Network Tools**

### **Test Connectivity: ping**

```

ping google.com

```

Output:

```

PING google.com (142.250.80.46) 56(84) bytes of data. 64 bytes from 142.250.80.46: icmp_seq=1 ttl=117 time=12.3 ms

```

Press `Ctrl+C` to stop.

```

ping -c 4 google.com

```

Send only 4 pings and stop.

### **Download Files: curl**

```

curl [https://example.com](https://example.com)

```

Displays the content (HTML).

```

curl -O [https://raw.githubusercontent.com/mischavandenburg/lab/refs/heads/main/README.md](https://raw.githubusercontent.com/mischavandenburg/lab/refs/heads/main/README.md)

```

Downloads to a file (`-O` uses the remote filename).

```

curl -o myfile.md [https://raw.githubusercontent.com/mischavandenburg/lab/refs/heads/main/README.md](https://raw.githubusercontent.com/mischavandenburg/lab/refs/heads/main/README.md)

```

Downloads with a custom filename.

```

curl -I [https://example.com](https://example.com)

```

Show just the headers (useful for checking if a site is up).

### **Download Files: wget**

```

wget [https://raw.githubusercontent.com/mischavandenburg/lab/refs/heads/main/README.md](https://raw.githubusercontent.com/mischavandenburg/lab/refs/heads/main/README.md)

```

Downloads to a file. Simple and reliable.

### **View Network Connections: ss**

```

ss -tuln

```

Output:

```

Netid State Recv-Q Send-Q Local Address:Port Peer Address:Port tcp LISTEN 0 128 0.0.0.0:22 0.0.0.0:*

```

Flags:

-   `-t` - TCP
    
-   `-u` - UDP
    
-   `-l` - Listening (server) sockets
    
-   `-n` - Show port numbers (not service names)
    

This shows [SSH](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) (port 22) is listening.

See established connections:

```

ss -tun

```

## **10.3 SSH Deep Dive**

### **What is SSH?**

SSH ([Secure Shell](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)) creates an encrypted connection between your computer and a remote server.

Everything you type and all output is encrypted.

Components:

-   **[SSH](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) client** - On your computer (you type `ssh`)
    
-   **SSH server** - On the remote machine (sshd daemon)
    
-   **Encryption** - Your traffic is unreadable to anyone in between
    

### **How SSH Works**

1.  You run `[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) user@server`
    
2.  Client connects to server port 22
    
3.  They negotiate encryption (key exchange)
    
4.  You authenticate (password or key)
    
5.  Encrypted channel established
    
6.  Your commands run on the server
    

### **Password vs Key Authentication**

**Password authentication:**

-   Simple but less secure
    
-   You type password every time
    
-   Vulnerable to brute force attacks
    

**Key authentication:**

-   More secure
    
-   No password needed once set up
    
-   Based on cryptographic key pairs
    
-   Required by many cloud providers
    

### **Generate SSH Keys**

```

ssh-keygen -t ed25519 -C "[your_email@example.com](mailto:your_email@example.com)"

```

Output:

```

Generating public/private ed25519 key pair. Enter file in which to save the key (/home/user/.[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/id_ed25519): Enter passphrase (empty for no passphrase):

```

Press Enter to accept the default location. You can add a passphrase for extra security.

This creates:

-   `~/.[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/id_ed25519` - Private key (NEVER share this)
    
-   `~/.ssh/id_ed25519.pub` - Public key (safe to share)
    

### **Copy Key to Server**

```

ssh-add ~/.[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/id_ed25519

ssh-copy-id user@192.168.100.71

```

You’ll enter your password once. After this, you can log in without a password.

Manual method (if `ssh-copy-id` isn’t available):

```

cat ~/.[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/id_ed25519.pub | ssh user@192.168.100.71 "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"

```

### **Verify Key Authentication is Working**

After copying your key, verify it’s actually being used. Use verbose mode:

```

[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) -v user@192.168.100.71

```

Look for these lines in the output:

```

debug1: Authentications that can continue: publickey,password debug1: Offering public key: /home/user/.[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/id_ed25519 ED25519 debug1: Server accepts key: /home/user/.ssh/id_ed25519 ED25519 debug1: Authentication succeeded (publickey).

```

The key lines are:

-   `Offering public key` - [SSH](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) is trying your key
    
-   `Server accepts key` - The server recognized your public key
    
-   `Authentication succeeded (publickey)` - You’re in with your key, not password
    

If you see `Authentication succeeded (password)` instead, your key isn’t set up correctly.

### **Disable Password Authentication**

Once key authentication works, disable password login for better security. This prevents brute force attacks.

On the **server** (the machine you [SSH](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) into):

```

sudo vi /etc/ssh/sshd_config

```

Find and change these lines:

```

PasswordAuthentication no PubkeyAuthentication yes

```

If the lines have `#` at the start, remove the `#` to uncomment them.

Restart the [SSH](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) service:

```

sudo systemctl restart sshd

```

**Test before disconnecting:** Open a new terminal and verify you can still connect:

```

[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) user@192.168.100.71

```

If key auth fails now, you’ll be locked out. That’s why you test with a separate connection while keeping your current session open.

### **SSH Config File**

Create `~/.[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/config`:

```

vi ~/.ssh/config

```

Add:

```

Host cato HostName 192.168.100.71 User yourusername IdentityFile ~/.[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/id_ed25519

```

Now instead of:

```

[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) yourusername@192.168.1.105

```

You just type:

```

[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) cato

```

Set proper permissions:

```

chmod 600 ~/.[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/config

```

### **Copy Files: scp**

```

[scp](https://app.getrecall.ai/item/e9163362-458b-4571-bc7f-2a8baa23d3dd) file.txt user@server:/home/user/

```

Copy local file to remote server.

```

[scp](https://app.getrecall.ai/item/e9163362-458b-4571-bc7f-2a8baa23d3dd) user@server:/var/log/syslog ./

```

Copy remote file to local directory.

```

[scp](https://app.getrecall.ai/item/e9163362-458b-4571-bc7f-2a8baa23d3dd) -r directory/ user@server:/home/user/

```

Copy a directory recursively.

### **SSH Best Practices**

1.  **Use keys, not passwords** - Disable password auth if possible
    
2.  **Use a passphrase** - Protects your key if someone gets the file
    
3.  **Use the config file** - Simpler commands, fewer mistakes
    
4.  **Keep private keys private** - Never share, never commit to git
```

```
## **Summary**

-   `ip addr` shows network interfaces, `ip route` shows routing
    
-   `/etc/hosts` maps hostnames to IPs locally
    
-   `ping` tests connectivity, `curl`/`wget` download files
    
-   `ss -tuln` shows listening ports
    
-   [SSH](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) keys are more secure than passwords
    
-   `ssh-keygen` creates keys, `ssh-copy-id` installs them
    
-   `ssh -v` shows which authentication method is used
    
-   Disable password auth in `/etc/[ssh](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)/sshd_config` after key auth works
    
-   `~/.ssh/config` simplifies SSH commands
    
-   `scp` copies files securely between machines

## **Definitions**

**IP Address**: A numerical identifier for a device on a network.

**Gateway**: The router that connects your network to others (usually the internet).

**SSH**: [Secure Shell](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54) - encrypted protocol for remote access.

**SSH Key Pair**: A public and private key for authentication without passwords.

**scp**: [Secure copy](https://app.getrecall.ai/item/e9163362-458b-4571-bc7f-2a8baa23d3dd) - transfers files over SSH.

**Port**: A number identifying a specific service ([SSH](https://app.getrecall.ai/item/5b0a7572-3af3-4ed7-a255-fcf2ac597b54)=22, HTTP=80, HTTPS=443).

**Loopback**: The 127.0.0.1 address that refers to the local machine.

**sshd\\_config**: The SSH server configuration file at `/etc/ssh/sshd_config` that controls how the SSH daemon accepts connections.
```
